# OkJson 产品转向设计:从树形浏览器到 JSON 文本编辑器

- **日期**: 2026-06-24
- **状态**: 已评审通过(待落地)
- **范围**: 产品方向转向 + 对应的架构重构

---

## 1. 背景与动机

### 现状

当前 OkJson 本质是一个**只读的树形 JSON 浏览器**:粘贴 JSON 后渲染成可折叠的树(`UnifiedJsonViewController`)。用户**不能**像在文本里那样随手点光标、改一个值、框选一段复制——要复制只能右键选节点的 Key/Value,要"改"只能删节点。

### 痛点

- **编辑麻烦**:它不是编辑器,是只读树 + 右键操作。
- **复制麻烦**:复制要靠右键菜单选节点,无法自由框选任意文本片段。
- **整体太复杂**:树形渲染 + 懒加载 + 节点菜单 + 多列等堆叠,概念和交互负担重。

### 转向

参照 Sublime Text 的体验:主体就是一块**可自由编辑、自由选择、自由复制**的文本(带 JSON 语法高亮、行号、一键格式化、代码折叠),简单直接。唯一要比 Sublime 多出来的,是它给不了的**联动对比(同步滚动)**。

---

## 2. 产品定位(一句话)

**一个纯文本 JSON 编辑器(语法高亮 + 行号 + 自由编辑/选择/复制 + 代码折叠 + 自动格式化),支持多栏并排 + 同步滚动,让用户用自己的眼睛对比。**

---

## 3. 核心决策摘要

| 决策点 | 结论 |
|---|---|
| 界面形态 | **纯文本编辑器 + 代码折叠**,删除独立树形视图。折叠替代"树形浏览" |
| 栏数 | **保留任意多栏**(`⌘D` 加列),不限制用户 |
| 联动对比 | **只做同步滚动**,工具**不判定差异**(避免规则与用户预期不符,如 `1.23` vs `1.234`) |
| 同步折叠 | **不做**(判定为伪需求,YAGNI;需要再说) |
| 编辑器内核 | **方案 A:纯原生自研**(`NSTextView` + TextKit),零第三方依赖,守住"轻量原生"定位 |
| 格式化触发 | 粘贴/打开**自动美化**;手敲编辑**不重排**(只校验高亮);`⌘R` 主动重排 |
| 非法 JSON | **不弹窗**,三层定位:行红三角 + 字符波浪线 + 底栏人话提示 |
| 差异判定 | 不做语义 diff、不做行级 diff、不自动对齐补空行 |
| 性能 | **硬指标**:大文件/复杂文件必须支持(见第 6 节) |

## 4. 架构与组件

### 4.1 层级结构

```
AppDelegate / MainWindowController / AppContainerViewController(底栏控件)   ← 基本不动
  └─ MainViewController(多列管理 + 同步滚动)            ← 保留,少量适配
       └─ JSONEditorViewController(每列 = 一个 JSON 文本编辑器)   ← 取代 FormatterVC + UnifiedJsonVC
            ├─ JSONTextView(NSTextView 子类:编辑/选择/复制/折叠交互/括号高亮)   ← 新增
            ├─ LineNumberRulerView(复用 + 增强:折叠三角、出错行红三角)
            └─ ColumnHeaderView(复用:标题/颜色/关闭)
```

一句话概括:**把"每列里的树形渲染器"整个换成"一个 JSON 文本编辑器",外层的多列骨架和同步滚动原封不动。**

### 4.2 Services / Model

| 组件 | 处理 |
|---|---|
| `IndexedJSONNode` | ♻️ 复用——记录的字节范围,喂给折叠区间与错误定位 |
| `JSONParser` | 🔧 增强——成功返回结构树;失败返回**精确位置 + 错误分类 + 人话原因** |
| `JSONFormatter` | ♻️ 复用——缩进 2/4、Key 排序的美化输出 |
| `JSONSyntaxHighlighter` | ➕ 新增——只对**可见视口**着色 |
| `FoldingModel` | ➕ 新增——从结构树算出可折叠区间 + 维护折叠状态 |
| `ClipboardService` / `CopyToastView` | ♻️ 复用(`CopyToastView` 用于"复制格式化结果"反馈) |

### 4.3 改造点

- **智能列宽 `⌘⇧W`**:现依赖树形的内容宽度估算(`estimatedContentWidth()`),改为**按最长行**估算宽度。
- **同步滚动**:`MainViewController` 已实现(按垂直位置同步、多列联动、防递归锁),只需把每列的 `mainScrollView` 指向新编辑器的 scrollView。
- **多列 Paste**:`MainViewController.keyDown` 已把 `⌘V` 路由到焦点列,沿用。

## 5. 数据流

核心原则:**重活全在后台线程,主线程只负责"应用结果 + 绘制"**。每列一个状态源,持有〔当前文本 · 最近一次成功解析的结构 · 折叠状态 · 错误信息〕。

### 路径 A — 粘贴 / 打开文件(会自动美化)

```
⌘V / 打开 → 原始文本入编辑器
   └─[后台] JSONParser 解析
        ├─ 成功 → JSONFormatter 美化(缩进 + Key 排序)
        │         └─[主线程] 美化文本上屏 → 重新解析定位 → 建折叠区间 → 视口着色
        └─ 失败 → 保留原始文本 →[主线程] 标错(红三角 + 波浪线 + 底栏提示)
```

### 路径 B — 手敲编辑(实时校验,绝不重排)

```
敲键 → 文本变化 → 防抖(停顿 ~300ms)
   └─[后台] 解析当前文本
        ├─ 成功 → 更新结构 / 折叠点,清除错误
        └─ 失败 → 标错;文本和光标原样不动
   （全程不自动美化;要美化按 ⌘R 主动触发 → 走路径 A 的美化分支）
```

### 其余联动

- **滚动** → 视口变化 → 只对**新进入视口的范围**着色(增量);若开同步滚动 → 当前列位置同步给其余列。
- **折叠** → 点 gutter 三角 → `FoldingModel` 切换该区间 → `JSONTextView` 隐藏/展开对应文本 → 行号与布局刷新。
- **非法 → 改回合法** → 错误自动清除、折叠恢复可用,但**不主动重排**(保住光标)。

## 6. 性能策略(硬指标)

关键判断:**这次重构对性能是利好**。旧树形要为每个节点建视图和布局,JSON 越大越复杂负担越重(所以被迫加懒加载)。换成文本编辑器后,开销只和**屏幕可见行数**相关,基本和 JSON 总大小、嵌套深度**解耦**。

### 五条策略

1. **视口着色(最关键)**:绝不全量着色。只给当前可见区 + 上下各一屏缓冲做语法高亮,滚动时增量补。
2. **后台解析 + 防抖**:解析/美化/排序全在后台线程,编辑防抖 ~300ms。`IndexedJSONNode` 零拷贝索引解析低内存、快。
3. **粘贴大文件先美化再上屏**:压缩成一行的 JSON(单行几 MB)是文本布局杀手。超长内容不直接上屏,后台先美化拆成多行再上屏,期间底栏显示"正在格式化…"。
4. **折叠即性能手段**:折叠大容器后,需布局的可见行数骤降。超大容器(上万元素)可默认折叠。
5. **着色分块应用**:属性分块写入,不一次性大范围 `setAttributes` 卡主线程。

### 性能目标(作为验收标准)

| 场景 | 目标 |
|---|---|
| 打开 / 粘贴 10 MB JSON | 后台美化 ≤ 2–3 s,期间 UI 不卡(有 loading) |
| 滚动浏览 | 稳定 60 fps |
| 手敲编辑 | 按键无可感延迟 |
| 折叠 / 展开超大容器 | 即时响应 |

### 风险点(实现期需原型验证)

- **代码折叠的底层实现**(TextKit 隐藏字形方式)是最不确定的技术点,先做最小原型验证可行性与性能,再铺开。
- **超长单个字符串值**(如一个值就是几 MB 的 base64)美化也拆不开。兜底:对这类超长行开启**软换行**,避免超长单行拖垮布局。

## 7. 错误处理

### 非法 JSON——三层定位

1. **行级** · 出错行 gutter 亮红三角 + 整行淡红底(一眼扫到大概位置)
2. **字符级** · 出错字符下红色波浪线(精确到那个符号)
3. **人话** · 底栏红色提示条「第 X 行:对象里缺少逗号 / 这里该是 `}`」,点击 → 光标跳到出错处

### 对 `JSONParser` 的要求

报出〔出错字节偏移 → 行/列〕+ 错误分类(缺逗号 / 缺冒号 / 括号或引号未闭合 / 非法转义 / 多余逗号 / 意外结束 EOF),映射成中文短句(复用并扩展 `Constants.ErrorMessages`)。

### 配套行为

- 非法期间保留输入原样,不擅自改动。
- 格式化与折叠在非法时暂停(无完整结构无法折叠)。
- 改回合法 → 错误清除、折叠恢复。

### 边界场景

| 情况 | 处理 |
|---|---|
| 空输入 | 不算错,显示干净空状态 + 占位提示「粘贴 JSON 开始」 |
| 多处错误 | 只报**最靠前的第一个**(解析器在首错处停),改完再报下一个 |
| 粘进非 JSON 文本 | 当非法 JSON 正常走错误流程 |
| 文件 > 10 MB | 打开时拦截,底栏提示超限(保留现有上限) |
| 打开失败 / 剪贴板空 | 底栏轻提示,**不弹窗** |

原则:**所有错误用"行内标记 + 底栏提示"表达,全程不弹模态框打断。**

## 8. 测试策略

务实原则:**能抽成纯逻辑的尽量单测,AppKit UI 交互用手动验收清单兜底**,不假装做全套 UI 自动化。

### 8.1 核心逻辑单元测试

- `JSONParser` 增强:每种错误类型一个用例(缺逗号 / 缺冒号 / 括号未闭合 / 引号未闭合 / 非法转义 / 多余逗号 / EOF / 空输入 / 非 JSON 文本),断言**位置和分类正确**。
- `FoldingModel`:给定 JSON,折叠区间(起止、嵌套关系)正确;覆盖嵌套对象/数组、空容器、深层嵌套。
- `JSONFormatter`:缩进 2/4、Key 排序、**幂等性**(格式化两次结果一致)。
- 字节偏移 ↔ 行列换算:含多字节 UTF-8 字符(中文、emoji)不算错位置。

### 8.2 性能测试

- 准备 1 MB / 10 MB、深层嵌套、超多元素数组、超长字符串值样本;用 `measure` 块测解析+美化耗时;**验证视口着色开销不随文件增大上升**。

### 8.3 可抽离的交互逻辑

- 同步滚动"源列位置 → 目标列位置"映射抽成纯函数单测;视口着色"该给哪些范围上色"做逻辑测试,不依赖真实渲染。

### 8.4 清理旧测试

- 删除/重写所有引用已删树形 API(`UnifiedJsonViewController`、懒加载、节点菜单)的旧用例,让重构后 `swift test` 回到**真实全绿**。

### 8.5 手动验收清单

粘贴 10 MB 不卡 / 滚动跟手 / 手敲不卡 / 折叠即时 · 非法 JSON 三层标记 + 点击跳转 · 多栏同步滚动 · 任意选区复制 / `⌘R` 重排 / 主题切换 / 打开 `.json` `.xcs`。

## 9. 删除 / 保留 / 新增 总账

### 🗑 删除

- `UnifiedJsonViewController`(树形渲染器,整块)
- `LargeValuePopover`(大值弹窗,树形特性)
- 懒加载机制(>50 子项按需加载)
- 节点右键菜单(复制 Key/Value/Key-Value、删除节点)
- "选中节点高亮匹配括号"(改为光标贴括号时高亮配对)

### ♻️ 保留 / 平移

语法高亮、行号、缩进 2/4、Key 排序、亮/暗主题、多栏 + 列头、同步滚动、自适应列宽(换实现)、打开 `.json` 与 `.xcs`、自动更新、10 MB 上限、`CopyToastView`。

### ➕ 新增

- `JSONTextView`(NSTextView 子类)
- `JSONSyntaxHighlighter`(视口着色)
- `FoldingModel`(折叠区间 + 状态)
- 代码折叠交互(gutter 三角)
- 实时 JSON 校验 + 三层错误定位
- 光标括号配对高亮

## 10. 风险与未决

| 项 | 说明 | 处理 |
|---|---|---|
| 代码折叠底层实现 | TextKit 隐藏字形/布局的可行性与性能 | 实现期先做最小原型验证 |
| TextKit 1 vs 2 | macOS 13+ 默认 TextKit 2,折叠/RulerView 配合可能有坑 | 实现期实测选定,必要时局部回退 |
| 超长单字符串值 | 美化拆不开 | 软换行兜底 |
| 解析器错误粒度 | 现有 `JSONParser` 报错信息是否够精细 | 需增强以输出精确位置 + 分类 |

## 11. 验收标准

1. 主界面是纯文本编辑器,可自由点光标、框选、`Cmd+C` 复制任意片段、直接改值。
2. JSON 语法高亮、行号、gutter 折叠三角可用;折叠/展开按 `{}` `[]` 层级正确。
3. 粘贴/打开自动美化;手敲不重排;`⌘R` 主动重排。
4. 非法 JSON 三层标记正确显示,底栏提示可点击跳转;全程无模态弹窗。
5. 多栏并排,`⌘D` 加列,同步滚动开关可用且多栏联动。
6. 性能目标(第 6 节表格)全部满足。
7. `swift test` 真实全绿;`swiftlint lint --strict` 通过。
